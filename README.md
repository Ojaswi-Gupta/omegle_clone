# Mini Omegle Clone

A web-based video chat application inspired by Omegle, built with modern web technologies. This project allows users to connect anonymously with strangers via peer-to-peer video and audio calls.

**🌍 Live Demo:** [https://omegle-clone-silk.vercel.app](https://omegle-clone-silk.vercel.app)

## Features

- **Anonymous Video Chat**: Instantly connect with random strangers.
- **Peer-to-Peer Communication**: Real-time, low-latency video and audio streaming using WebRTC.
- **Firebase Signaling**: Robust connection negotiation (offers, answers, ICE candidates) handled via Firebase Firestore.
- **Anonymous Authentication**: Frictionless user entry using Firebase Anonymous Auth.
- **Next/Skip functionality**: Easily skip to the next available stranger.

## System Architecture

The application relies on a decentralized, peer-to-peer architecture for real-time video streaming, with a central signaling server to coordinate connections:

1. **Client (Next.js)**: Handles the UI, media streams (camera/microphone), and WebRTC peer connection logic.
2. **Signaling Server (Firebase Firestore)**: Acts as the intermediary for matchmaking and WebRTC signaling. It facilitates the exchange of Session Description Protocol (SDP) offers/answers and ICE candidates between peers before a direct connection is established.
3. **STUN/TURN Servers (Metered)**: Used for NAT traversal. STUN servers help clients discover their public IP addresses, while TURN servers relay media if a direct P2P connection fails.

## Database Schema (Firestore)

The Firestore database relies on the following collections and document structures to handle matching and signaling:

- **`waiting`** (Collection)
  - `createdAt` (Timestamp): Used to find the earliest waiting user.
  - *Note*: Documents here represent users waiting for a match. When a second user joins, they delete the document and start the signaling process using its ID as the room ID.

- **`rooms`** (Collection)
  - `createdAt` (Timestamp)
  - `offer` (Object): The WebRTC SDP offer generated by the caller.
  - `answer` (Object): The WebRTC SDP answer generated by the callee.
  - **`callerCandidates`** (Sub-collection)
    - Documents containing the caller's ICE candidates.
  - **`calleeCandidates`** (Sub-collection)
    - Documents containing the callee's ICE candidates.

## Application Workflow

1. **Matchmaking Flow**:
   - When a user clicks **Start**, they query the `waiting` collection.
   - **If the collection is empty**: The user creates a new document in `waiting` and assumes the role of **Caller**.
   - **If a document exists**: The user retrieves it, deletes it from `waiting` to prevent others from joining, and assumes the role of **Callee**. The ID of the deleted document is used as the `room` ID.

2. **WebRTC Signaling Flow**:
   - **Caller**: Creates an RTCPeerConnection, generates an `offer`, and saves it to the `rooms/{roomId}` document. They then listen for an `answer` field to be populated.
   - **Callee**: Retrieves the `offer` from `rooms/{roomId}`, sets their remote description, generates an `answer`, and updates the same document with it.
   - **Both**: As the `RTCPeerConnection` generates ICE candidates locally, they push them to their respective sub-collections (`callerCandidates` or `calleeCandidates`). Concurrently, they listen to the other party's sub-collection and add any new remote ICE candidates to their peer connection.

3. **Media Flow**:
   - Once signaling is complete, the `RTCPeerConnection` establishes a direct P2P connection (or relays via TURN) and media streams are exchanged directly between browsers.

## Tech Stack

- **Framework**: [Next.js 16](https://nextjs.org/) (App Router)
- **UI Library**: [React 19](https://react.dev/)
- **Language**: [TypeScript](https://www.typescriptlang.org/)
- **Styling**: [Tailwind CSS v4](https://tailwindcss.com/)
- **Backend/Signaling**: [Firebase](https://firebase.google.com/) (Firestore & Auth)
- **Real-time Communication**: [WebRTC](https://webrtc.org/) (with STUN/TURN servers provided by Metered)

## Prerequisites

Before running the project, make sure you have the following installed:

- Node.js (v20+ recommended)
- npm, yarn, or pnpm
- A [Firebase Project](https://console.firebase.google.com/) with Firestore Database and Anonymous Authentication enabled.

## Setup & Installation

1. **Clone the repository**:
   ```bash
   git clone <repository-url>
   cd omegle_clone/frontend
   ```

2. **Install dependencies**:
   ```bash
   npm install
   # or
   yarn install
   # or
   pnpm install
   ```

3. **Configure Firebase**:
   Create a `.env.local` file in the `frontend` directory and add your Firebase configuration:
   ```env
   NEXT_PUBLIC_FIREBASE_API_KEY=your_api_key
   NEXT_PUBLIC_FIREBASE_AUTH_DOMAIN=your_auth_domain
   NEXT_PUBLIC_FIREBASE_PROJECT_ID=your_project_id
   NEXT_PUBLIC_FIREBASE_STORAGE_BUCKET=your_storage_bucket
   NEXT_PUBLIC_FIREBASE_MESSAGING_SENDER_ID=your_messaging_sender_id
   NEXT_PUBLIC_FIREBASE_APP_ID=your_app_id
   ```

   **Note on Firebase Setup:**
   - Go to the Firebase Console, create a project, and add a Web App.
   - Go to **Authentication** -> **Sign-in method** and enable **Anonymous**.
   - Go to **Firestore Database** and create a database. Update your security rules appropriately to allow reads/writes for testing. 

## Running the Application

Start the development server from the `frontend` directory:

```bash
npm run dev
# or
yarn dev
```

Open [http://localhost:3000](http://localhost:3000) with your browser to see the application in action.

*Note: For the camera access and WebRTC connections to work properly across different devices, the site must be served over HTTPS (or `localhost` for local development).*
